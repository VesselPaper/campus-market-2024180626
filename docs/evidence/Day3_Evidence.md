# Day3 Evidence - 首页展示优化与消息修复

## 1. 今日完成内容

今天主要完成了首页展示优化、消息系统问题修复和用户交互体验增强，主要包括以下工作：

首页方面，去掉了之前的欢迎横幅和全部信息列表（商品信息转移到集市展示），新增了一个自动轮播的 Banner 推荐大屏，按浏览量排序展示热门商品和平台公告。轮播每 4 秒自动切换，鼠标悬停暂停，底部的圆点导航可以手动切换。轮播下方保留了逛逛集市入口、快速操作按钮和推荐商品卡片。

集市页面去掉了底部分页器（ElPagination），所有商品直接展示出来，用户只需向下滚动就可以浏览全部信息。同时在右下角增加了一个回到顶部按钮，滚动超过 300px 时自动出现，点击平滑回到页面顶部。

消息功能方面，修复了两个问题：一是未读红点把用户自己发送的消息也算进去了，现在只有别人回复的消息才会计入未读数；二是自己给自己发消息也会生成自动回复，现在添加了判断，只有发送者和接收者不是同一人时才会生成回复。

右上角的用户头像改为下拉菜单，鼠标悬停显示：个人中心、消息中心（带未读红点）、浏览记录、账号设置和退出登录。原来的独立退出按钮已移除，统一整合到下拉菜单中。

"我的"页面（ProfileView）新增了时间段问候语（上午好/下午好等），将原本在首页的欢迎横幅移到了个人中心。

新增了浏览记录（HistoryView）和账号设置（SettingsView）两个页面。浏览记录通过 localStorage 自动保存用户查看过的商品，最多保留 50 条，支持清空。账号设置页面展示了用户信息表单和注销账号功能。

## 5. 遇到的问题与解决方法

**问题：JSON Server 在发送多条消息后崩溃。**

在测试消息功能时，发送多条消息后 json-server 后端会报 `EADDRINUSE` 错误并崩溃。查看日志发现 `npm run mock` 使用了 `--watch` 参数，每当 POST/PATCH 请求写入 `db.json` 文件时，json-server 会检测到文件变更并触发重载。重载过程中旧进程还未完全释放端口，新进程尝试监听同一个端口就导致了 `EADDRINUSE` 冲突，最终进程退出。

**解决方法**：去掉 `--watch` 参数，直接用 `npx json-server db.json --port 3000` 启动。这样 json-server 不再监听文件变化，完全在内存中运行，不会因为文件写入触发重载导致端口冲突。注意这种模式下数据变更不会自动写回 `db.json`（快照需要手动按 `s` 键），但对开发调试没有影响。

## 6. 今日反思

通过今天的任务，我对前端项目中的几个常见问题有了更深的理解。

第一个是未读数计算的归属问题。在消息系统中，未读数的含义是"用户还没看过的消息数"。用户自己发送的消息显然不应该算作未读，只有对方发来的消息才需要计入。最初的设计简单地在每次发消息时都 +1，没有区分消息方向，导致红点数量翻倍。这提醒我在设计数据逻辑时，一定要想清楚每个数据的语义是什么、谁应该看到它。

第二个是后端稳定性对开发效率的影响。json-server 的 `--watch` 模式虽然方便（自动同步文件），但在高频率写入时容易出现端口冲突崩溃。看似只是一个启动参数的问题，但实际排查时花费了不少时间。后续可以考虑改用更稳定的 mock 方案，或者自己写一个简单的 Express mock 服务。

第三个是轮播组件的实现细节。自动轮播看似简单，但有几个要点需要注意：切换动画要用 opacity + transform 而非 display 控制，避免闪烁；导航圆点的当前状态要清晰可辨；鼠标悬停暂停是一个重要的用户体验细节，防止用户在阅读时突然切换走。把常用组件（BackToTop、BannerCarousel）封装成独立组件也有利于后续复用。

总的来说，今天的任务覆盖了功能优化、bug 修复和新功能开发三种类型的工作，让我体会到一个项目在日常维护中可能面临的各种情况。
